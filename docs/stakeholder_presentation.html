Slide 1

Turn the experience study into a scalable, auditable workflow.

The business goal is straightforward: reduce the time, specialist bottlenecks, and spreadsheet risk involved in traditional experience-study work without giving up actuarial control.

This system does that by wrapping deterministic data-science workflows in a constrained agent interface, so more people can safely run the process while still tracing every quantitative output back to code and artifacts.

Scaling Expertise Natural Language More actuaries can trigger profiling, sweeps, and reporting without being Python specialists.
Operational Efficiency One Guided Flow Profiling, analysis, and reporting happen in one session with less manual handoff time.
Governance Auditable Artifacts The workflow leaves behind prepared datasets, sweep outputs, and HTML reports for review.
Business payoff, tied to architecture

Less specialist bottleneck

Users get a guided interface instead of a collection of ad hoc scripts and manual steps.

Less spreadsheet risk

Validation rules, A/E math, and report generation live in deterministic Python rather than human copy-paste workflows.

What makes this credible

The business story only works because the architecture is constrained: the LLM routes and narrates, while the tool layer owns the actual actuarial work.

Slide 2

The architecture makes the business value operationally real.

This is not a generic assistant with broad access. It is a constrained orchestration engine: one supervisor, three specialist agents, schema-bound tool calls, and explicit handoffs between deterministic stages.

1 Orchestrator Converts natural language into one controlled execution path.
3 Specialists Steward, Actuary, and Analyst each own a narrow job boundary.
Schema Contracts Tool arguments are constrained before execution.
Deterministic Tools Python and DuckDB own the calculations and file writes.
Artifacts Prepared data, sweep outputs, and reports remain inspectable on disk.
scale expertise safely reduce context switching preserve audit trail reusable blueprint

Slide 3

Each specialist agent owns a narrow slice of the workflow.

The multi-agent design works because the agents are constrained by role and by tool surface. Each one can do only a small number of things, and those things map directly to files in the repo.

Data Steward Agent Input trust and feature engineering

Reads source data, profiles structure, validates actuarial data quality, and writes the prepared analysis dataset.

profile_dataset run_actuarial_data_checks create_categorical_bands regroup_categorical_features
Lead Actuary Agent Dimensional analysis and ranked interpretation

Calls the sweep engine, compares count and amount A/E, and summarizes the top-ranked cohorts without performing arithmetic itself.

run_dimensional_sweep selected_columns top_n sort_by
Analyst Agent Visualization and communication layer

Reads sweep summaries and generates browser artifacts. It never visualizes raw inforce data directly.

generate_combined_report metric=count|amount
Study Orchestrator Session state and handoff logic

Preserves workflow continuity, owns pending follow-up prompts, and decides which specialist gets the next turn.

_classify_intent _handle_continue _route_to_steward _route_to_actuary _route_to_analyst
Slide 4

Tooling is explicit: agents call a constrained set of deterministic functions.

1
Schemas

Pydantic defines the call contract

ProfileDatasetSchema, DimensionalSweepSchema, and VisualizationSchema become OpenAI tool specs.

2
Handlers

Agents map tool names to local functions

Each agent has a small tool_handlers dictionary rather than broad code execution freedom.

3
Execution

Python tools own computation and file writes

The actual work is done in tools/, and the results are written to predictable artifacts.

4
Narration

Agents convert tool output into operator language

The model explains what happened, but it does not become the source of truth.

DataStewardAgent -> profile_dataset -> run_actuarial_data_checks -> create_categorical_bands -> regroup_categorical_features ActuaryAgent -> run_dimensional_sweep AnalystAgent -> generate_combined_report
Slide 5

The deterministic tool layer is where the hard guarantees live.

Once the architecture is visible, the important question becomes: what can each tool actually do, and what does the system guarantee about those outputs?

Data Steward tool

Profiles types, validates MAC/MEC/MOC and COLA logic, and writes analysis_inforce.parquet without overwriting raw input.

Insight Engine tool

Computes DuckDB-backed dimensional sweeps, A/E ratios, and Bayesian confidence intervals with deterministic Python and SciPy.

Visualization tool

Reads aggregated sweep outputs and writes a combined A/E report with forest plot, detail table, and treemap.

Agent boundary

The LLM chooses, invokes, and summarizes tools. It does not invent formulas, alter source truth, or compute actuarial metrics itself.

Repo-level guarantees already encoded in tools
MOC rules 0 < MOC <= 1.0 MAC == 1 => MOC == 1.0 Bayesian credibility prior = Beta(0.5, 0.5) Amount fallback if MAC > 0: avg_claim = MAF / MAC else: avg_claim = MEF / MEC
Technical meaning

These are engineering constraints implemented in Python, not presentation claims layered on top of a generic assistant.

Slide 6

Artifacts are how the agent architecture stays inspectable.

The most useful architectural detail for technical reviewers is that handoffs do not live only in memory. They land on disk as concrete files that the next agent consumes.

Steward output analysis_inforce.parquet Prepared dataset used by the actuary tool layer as the analysis source of truth.
Actuary output sweep_summary.csv Ranked cohort results plus flattened CI columns required by visualization.
Analyst output HTML reports Browser-openable reports created from the latest sweep artifact, not from raw input.
Agent-to-agent handoff on disk
analysis_inforce.parquet produced by DataStewardAgent tools sweep_summary.csv produced by ActuaryAgent tools sweep_summary_1_gender_smoker.csv run-specific actuarial artifact combined_ae_report_<id>.html produced by AnalystAgent tools temp_treemap_source_gender_smoker.csv
Engineering benefit

When a step misbehaves, reviewers can inspect the exact artifact that crossed the boundary instead of reverse-engineering hidden model state.

Slide 7

Tests and UAT validate the agent boundaries, not just the outputs.

The strongest technical argument for this design is that the orchestration behavior is testable. The repo verifies routing decisions, artifact creation, and role discipline directly.

Orchestrator

Continuation and pending state

Tests verify that follow-up prompts advance through the staged workflow rather than being treated as generic chat replies.

Actuary Tooling

Deterministic sweep artifacts

Tests confirm that ranked CSVs are persisted, latest aliases are maintained, and CI columns are present for downstream tools.

Agent Contracts

Role and data-source discipline

Tests cover schema exposure, missing engineered columns, and the requirement for fresh sweep artifacts before visualization.

Automated signal

Current baseline: 23 passing tests covering orchestrator flow, insight engine behavior, and agent contracts.

Manual signal

docs/UAT_Checklist.md covers schema checks, session-stacking behavior, continuation routing, artifact creation, and graceful failures.
